# Stage 5 Narrative Contract

## Goal

Stage 5 starts by freezing the narrative-layer contract before any provider integration, prompt work, or report wiring happens.

The purpose is to keep the narrative layer narrow and grounded:

- rules remain the fact source
- LLM stays on the expression side
- fallback is explicit
- single-account and multi-account outputs use one narrative vocabulary

## Why Freeze the Contract First

The repo already has structured portrait facts:

- archetype
- tags
- evidence
- confidence
- stable traits
- community-specific traits
- overlap / divergence
- warnings and degraded signals

If MiniMax or any other provider is connected before the narrative input/output contract is frozen, the provider shape and prompt shape will start driving the architecture. This step prevents that.

## Core Terms

### `NarrativeDraft`

- structured narrative output prepared for API/UI consumption
- not a prompt
- not a raw model response
- can be generated by rules only or by an LLM-assisted path

### `NarrativeSection`

- one structured section inside a narrative draft
- each section has a stable code, content, and grounding metadata

### `NarrativeMode`

- `OFF`
- `RULE_ONLY`
- `LLM_ASSISTED`

### `NarrativeTone`

- `NEUTRAL`
- `ANALYTICAL`
- `CONCISE`

### `NarrativeAudience`

- `PRODUCT_USER`
- `INTERNAL_QA`

### `NarrativeFallbackMode`

- `USE_RULE_SUMMARY`
- `SKIP_NARRATIVE`

### `LlmNarrativeGateway`

- provider-facing contract for structured narrative generation
- does not expose MiniMax SDK details
- does not own facts

## Minimal Fact Source for `ComposeNarrativeInput`

`ComposeNarrativeInput` is intentionally restricted to structured, rule-confirmed inputs:

- `portrait.archetype / tags / summary / confidence`
- `FeatureVector`
- `Signal[]`
- `stableTraits`
- `communitySpecificTraits`
- `overlap / divergence`
- `warnings`
- `degraded`
- `selectedEvidence`
- `accountCoverage`
- `mode / tone / audience / fallbackPolicy`

It does not accept:

- raw connector HTML
- raw connector response payloads
- frontend UI state
- provider-specific prompt fragments

## Boundary: Rules Own Facts, LLM Owns Expression

The frozen rule for Stage 5 is:

- archetype, tags, stable traits, community-specific traits, overlap/divergence, confidence, warnings, and degraded state come from the rules/report layer
- the narrative layer may reorganize and restate those facts
- the narrative layer must not invent new factual conclusions

This is especially important for:

- low-data cases
- degraded-source cases
- multi-community explanations

The future prompt builder must enforce conservative output when the structured input already says confidence is limited.

## What This Step Explicitly Does Not Do

- no real provider integration
- no MiniMax gateway
- no prompt builder
- no report integration
- no UI changes

## How Later Steps Use These Contracts

### Step 2

- implement `MiniMaxNarrativeGateway` and provider abstraction behind `LlmNarrativeGateway`

### Step 3

- build prompts from `ComposeNarrativeInput`
- generate structured `NarrativeDraft` sections

### Step 4

- decide how `NarrativeGenerationResult` is wired into `ReportBuilder`
- keep fallback explicit if LLM generation fails

### Step 5

- surface narrative sections in the result experience without changing the fact source
